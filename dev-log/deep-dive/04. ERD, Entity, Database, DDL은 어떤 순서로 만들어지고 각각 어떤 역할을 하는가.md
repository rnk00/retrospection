# 고민 주제

> **ERD, Entity, Database, DDL은 어떤 순서로 만들어지고 각각 어떤 역할을 하는 걸까?**

---

# 결론

ERD, Entity, DDL, Database는 각각 다른 역할을 하는 것이 아니라 **하나의 설계를 점점 실제 서비스로 구현해 가는 과정**이다.

**요구사항 → ERD → Entity → DDL → Database 생성**이라는 흐름으로 이해하면 헷갈리지 않는다.

---

# 배경

처음 백엔드 프로젝트를 하면 이런 생각이 든다.

> "ERD도 테이블 같은데 Entity도 있고 Database도 있고... 도대체 뭐가 다른 거지?"

헷갈리는 이유는 **모두 데이터와 관련된 것**이기 때문이다.

하지만 각각의 역할은 완전히 다르다.

| 역할       | 무엇인가?     |
| -------- | --------- |
| ERD      | 설계도       |
| Entity   | 자바 객체     |
| DDL      | DB 생성 명령어 |
| Database | 실제 저장 공간  |

---

# 동작 방식

## 1. 요구사항 분석

먼저 서비스에서 어떤 데이터를 관리할지 정한다.

예를 들어 게시판이라면 회원, 게시글, 댓글 이 필요하다.

---

## 2. ERD 작성

이제 데이터의 관계를 설계한다.

```
Member
-------
id
name

Post
-------
id
title
member_id

Comment
-------
id
content
post_id
```

여기서는

* Member는 Post를 여러 개 작성할 수 있다.
* Post는 Comment를 여러 개 가진다.

같은 **데이터 간 관계를 정의**한다.

즉, **ERD는 "데이터를 어떻게 저장할지"를 설계하는 단계이다.**

---

## 3. Entity 작성

ERD가 완성되면 이를 코드로 옮긴다.

예를 들어 Member 테이블은 Member Entity, Post 테이블은 Post Entity가 된다.

JPA에서는 Entity가 곧 테이블과 거의 1:1로 대응된다.

즉, ERD를 보고 Entity를 만드는 것이다.

---

## 4. DDL 생성

Entity가 준비되면 이제 실제 DB에 테이블을 만들어야 한다.

이때 사용하는 것이 DDL이다.

ex)CREATE TABLE, ALTER TABLE, DROP TABLE과 같은 SQL이다.

JPA는 Entity를 보고 자동으로 DDL을 생성할 수도 있다.

즉, Entity -> DDL 생성 -> CREATE TABLE ... 과정이 이루어진다.

---

## 5. Database 생성

DDL이 실행되면 MySQL이나 PostgreSQL 안에 실제 테이블이 만들어진다.

이제부터 사용자가 회원가입하면 Member 테이블에 데이터가 저장된다.

회원가입 -> Member Entity -> INSERT SQL -> Member Table -> Database 저장

---

6. 서비스 개발

# 많이 헷갈리는 부분

## ERD와 Entity는 같은 건가?

아니다.

ERD는 설계도이고, Entity는 그 설계도를 자바 코드로 구현한 것이다.

---

## Entity가 있으면 DB도 생기는 건가?

JPA 설정에 따라 다르다.

개발 환경에서는 Entity만 작성해도 Hibernate가 자동으로 DDL을 만들어 테이블을 생성해 준다.

하지만 운영 환경에서는 자동 생성을 거의 사용하지 않고, 직접 작성한 DDL이나 데이터베이스 마이그레이션 도구(Flyway, Liquibase 등)를 통해 테이블을 생성하는 경우가 많다.

---

## DDL은 누가 작성하는가?

두 가지 방법이 있다.

* 개발자가 직접 작성한다.
* JPA가 Entity를 보고 자동 생성한다.

실무에서는 테이블 변경 이력을 관리하기 위해 직접 관리하거나 마이그레이션 도구를 사용하는 경우가 많다.
