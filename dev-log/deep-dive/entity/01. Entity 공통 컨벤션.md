# 고민 주제

> **Entity 값을 어떻게 바꾸게 할까(Setter vs 도메인 메서드)? 기본 생성자는 열어둘까 막을까?**

---

# 결론

* 값 변경은 `@Setter`로 연다 (도메인 메서드 대신).
* 기본 생성자는 `protected`로 막고, 실제 생성용 생성자만 `public`으로 연다.

이 두 가지는 User뿐 아니라 **모든 Entity에 공통으로 적용**하는 컨벤션이다.

---

# @Setter vs 도메인 메서드

값을 바꾸는 방법은 두 가지가 있다.

**`@Setter`**: Lombok이 모든 필드에 대해 `setXxx()`를 자동 생성. 아무 필드나 코드 어디서든 바꿀 수 있음.

```java
user.setNickname("새 닉네임");
user.setEmail("아무값도 가능");
```

**도메인 메서드**: `changeNickname()`처럼 "이 값은 이 상황에서만 바뀐다"가 드러나는 메서드만 만듦.

```java
user.changeNickname("새 닉네임"); // 이런 메서드만 있고 setEmail() 자체가 없음
```

`@Setter`가 열려있으면 나중에 예상 못한 곳에서 값이 바뀌어서 버그 추적이 어려워질 수 있지만, 필드마다 메서드를 직접 만들어야 하는 도메인 메서드 방식보다 초반엔 손이 덜 감. **`@Setter`로 결정** — 실제로 문제가 생기면 그때 도메인 메서드로 전환하는 것도 방법.

---

# 기본 생성자를 protected로 막는 이유

JPA는 DB에서 데이터를 읽어올 때 리플렉션으로 파라미터 없는 생성자를 사용해서 객체를 만든다 (필드는 이후 리플렉션으로 직접 채움). 이 생성자가 `public`이면, 코드 어디서든 실수로 `new User()`를 호출해서 값이 하나도 없는 빈 껍데기 객체를 만들 수 있다.

그래서 `@NoArgsConstructor(access = AccessLevel.PROTECTED)`로 막아서, JPA만 이 생성자를 쓰고 실제 애플리케이션 코드는 값을 다 채우는 별도의 `public` 생성자만 쓰도록 강제한다.

```java
// 눈에 안 보임, Lombok이 자동 생성 (JPA 전용)
@NoArgsConstructor(access = AccessLevel.PROTECTED)

// 눈에 보임, 직접 작성 (실제 생성용)
public User(String email, String nickname, String provider, String providerUserId) { ... }
```

생성자에 어떤 필드를 받을지는 Entity마다 다르므로, 그건 각 Entity 문서(예: `User.md`)에 정리한다. 원칙은 **"이 시점에 실제로 알 수 있는 값만 받는다"** — 예를 들어 DB가 채번하는 `id`, 나중에 채워지는 값, JPA Auditing이 채우는 `createdAt`/`updatedAt` 등은 생성자에서 받지 않는다.

# 함께 알게 된 용어

### GenerationType.IDENTITY

PK 채번을 DB의 `AUTO_INCREMENT`에 맡기는 전략. MySQL을 쓸 때 일반적으로 선택하는 방식. 모든 Entity의 `@Id` 필드에 공통으로 사용.

# 한 줄 정리

> **Entity의 생성자/Setter 정책은 "누가, 언제, 어떻게 값을 바꿀 수 있게 할까"를 프로젝트 전체에서 일관되게 정하는 것이다.**
