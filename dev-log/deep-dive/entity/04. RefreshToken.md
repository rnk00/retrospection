# 고민 주제

> **로그인 유지를 어떻게 할까? Access Token만으로 충분할까, Refresh Token도 필요할까? Access Token과 달리 Refresh Token은 왜 DB에 저장해야 할까?**

---

# 결론

* Access Token(짧은 만료) + Refresh Token(긴 만료)을 같이 쓴다.
* Refresh Token은 DB 테이블(`RefreshToken`)로 관리해서, 로그아웃 시 진짜로 무효화할 수 있게 한다.
* 한 유저가 여러 기기에서 로그인할 수 있도록 `User : RefreshToken = 1:N`으로 설계한다.

---

# 배경

JWT 만료 시간을 어떻게 정할지 고민하다가, Access Token 하나만 쓰면 만료를 짧게 잡을 경우 자주 재로그인해야 해서 불편하고, 길게 잡으면 탈취됐을 때 위험 노출 기간이 길어지는 문제를 확인함.

---

# Access Token만 쓰기 vs Refresh Token도 쓰기

**Access Token만 사용**: 토큰 하나만 발급, 만료되면 다시 로그인(OAuth부터 다시).

**Refresh Token도 사용**: Access Token은 짧게(예: 1시간), Refresh Token은 길게(예: 2주) 만료를 잡아서, Access Token만 만료돼도 재로그인 없이 Refresh Token으로 새 토큰을 받음.

매일 쓰는 습관 형성 앱 특성상 자주 재로그인시키면 불편하다고 판단해서 **Refresh Token도 쓰는 걸로 결정**.

---

# Refresh Token은 왜 DB에 저장해야 할까

Access Token은 수명이 짧고 JWT 자체에 사용자 정보와 만료 시간이 포함되어 있어 서버가 서명만 검증하면 되므로 별도의 DB 저장이 필요 없다. 반면 Refresh Token은 새로운 Access Token을 발급할 수 있는 권한을 가지므로 서버에서 상태를 관리해야 한다. 따라서 DB(또는 Redis)에 저장하여 요청 시 저장된 토큰과 일치하는지 확인하고, 로그아웃 시 토큰을 삭제하거나, 탈취된 토큰을 무효화하거나, 사용자당 하나의 토큰만 유지하는 등의 보안 정책을 적용할 수 있다. 즉, Access Token은 인증을 위한 '검증 대상'이고, Refresh Token은 재발급 권한을 가진 '관리 대상'이기 때문에 DB에 저장한다.

---

# 기기별 다중 로그인

한 유저가 여러 기기(폰, PC 등)에서 동시에 로그인하는 경우를 고려함.

* `user_id`에 `UNIQUE` — 유저당 하나만 유지, 새로 로그인하면 이전 기기 로그인이 끊김
* `user_id`에 제약 없음 — 기기별로 독립적인 로그인 유지

한 기기에서 로그인해도 다른 기기 세션이 끊기지 않는 게 일반적인 사용자 기대에 맞아서 **후자로 결정**.

---

# User를 어떻게 참조할까: @ManyToOne vs Long userId

`RefreshToken`이 `User`를 참조하는 방법도 두 가지가 있었음.

**`@ManyToOne User user`**: 진짜 User 객체로 연결, `refreshToken.getUser()`로 탐색 가능. 대신 `auth` 패키지가 `user` 패키지의 `User` 클래스에 의존하게 됨.

**`Long userId`**: 숫자 값만 가짐. 도메인 간 결합은 약해지지만 객체 탐색은 불가능.

일반적인 Spring/JPA 프로젝트에서는 관계 있는 Entity끼리 `@ManyToOne`/`@OneToMany`로 객체 연결하는 게 더 흔하고 표준적인 방식이라 **`@ManyToOne`으로 결정**. 대신 `fetch = FetchType.LAZY`를 반드시 같이 써서, `user` 필드를 실제로 쓸 때만 User가 조회되게 함 (기본값인 EAGER로 두면 RefreshToken 조회할 때마다 불필요하게 User까지 매번 같이 불러오게 됨).

---

# 함께 알게 된 용어

### FetchType.LAZY vs EAGER

연관된 Entity(`User`)를 **언제 조회할지** 정하는 옵션.

* `LAZY`: `user` 필드에 실제로 접근하는 순간에만 DB 조회 (지연 로딩)
* `EAGER`: 연관관계 상관없이 항상 즉시 같이 조회 (즉시 로딩)

기본적으로 `@ManyToOne`은 `EAGER`가 기본값인데, 매번 불필요하게 User까지 같이 불러오는 걸 막기 위해 명시적으로 `LAZY`를 지정함.

---

# 함께 알게 된 위험 요소

`token` 값을 평문으로 저장하고 있는데, DB가 유출되면 이 값도 그대로 노출됨. PAT만큼 시급하진 않지만, 나중에 해싱 저장을 고려할 만한 지점으로 남겨둠.
