# 고민 주제

> **여러 Entity가 똑같이 필요로 하는 createdAt/updatedAt을, 매번 반복해서 써야 할까?**

---

# 결론

공통 부모 클래스 두 개(`BaseTimeEntity`, `BaseCreatedTimeEntity`)로 분리해서 상속받는다.

---

# 배경

User, Retrospective, KptNote는 `createdAt`+`updatedAt` 둘 다 필요하고, AiRecommendation/WeeklySummary/Summary/ProblemPattern은 `createdAt`만 필요함 (이력성 데이터라 수정이 없어서 `updatedAt` 자체가 없음).

---

# 반복 vs 상속

**반복하는 방식**: Entity마다 이 필드를 매번 따로 씀.

```java
class User {
    @CreatedDate
    private LocalDateTime createdAt;
    @LastModifiedDate
    private LocalDateTime updatedAt;
    // ... User만의 다른 필드들
}
```

**공통 부모 클래스 방식**: 이 필드를 담은 클래스를 만들어두고 상속받음.

```java
abstract class BaseTimeEntity {
    @CreatedDate
    private LocalDateTime createdAt;
    @LastModifiedDate
    private LocalDateTime updatedAt;
}

class User extends BaseTimeEntity {
    // createdAt/updatedAt은 안 써도 자동으로 있음
}
```

필요로 하는 Entity가 여러 개라 **상속**으로 결정. `updatedAt`이 필요 없는 4개 Entity를 위해 `BaseCreatedTimeEntity`(`createdAt`만 있는 버전)도 별도로 만들었다.

---

# 함께 알게 된 용어

### @MappedSuperclass

이 클래스 자체는 테이블이 되는 게 아니라, 상속한 Entity의 테이블에 필드(컬럼)를 끼워 넣어주는 용도라는 표시. 이게 없으면 JPA가 이 클래스를 어떻게 다뤄야 할지 모른다. `base_time_entity` 같은 별도 테이블은 생기지 않는다.

### @EntityListeners(AuditingEntityListener.class)

`@CreatedDate`/`@LastModifiedDate`는 "이 필드에 시간 넣어줘"라는 표시일 뿐이고, 실제로 저장/수정 시점을 감지해서 값을 채우는 동작은 이 리스너가 한다. 이게 없으면 어노테이션이 있어도 아무 일도 안 일어난다.

### @EnableJpaAuditing

앱 전체에서 JPA Auditing 기능 자체를 켜는 스위치. 메인 애플리케이션 클래스(`RetrospectiveApplication`)에 한 번만 붙이면 된다. 이게 없으면 각 클래스에 `@CreatedDate`/`@EntityListeners`를 다 붙여놔도 기능이 꺼진 상태라 아무것도 안 채워진다.

### 읽기 전용으로 둔 이유

Entity 공통 컨벤션에서는 `@Setter`를 열기로 했지만, `BaseTimeEntity`/`BaseCreatedTimeEntity`는 `@Getter`만 두고 Setter는 안 만들었다 — `createdAt`/`updatedAt`은 JPA Auditing이 자동으로 관리하는 값이라, 애플리케이션 코드가 직접 바꿀 이유가 없기 때문.

---

# 한 줄 정리

> **여러 Entity가 공통으로 필요로 하는 필드는, 매번 반복하기보다 부모 클래스로 뽑아서 상속받는 게 유지보수에 유리하다.**
