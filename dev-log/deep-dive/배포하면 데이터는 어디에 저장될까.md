# 배포 후 데이터의 저장 위치

서비스를 배포한 이후 사용자가 입력하는 데이터는 개발자의 개인 컴퓨터에 저장되지 않는다.
개발자의 노트북은 서비스를 만드는 공간일 뿐이고, 실제 운영 환경에서는 **24시간 실행되는 클라우드 서버 환경**에 데이터가 저장된다.

대표적으로 AWS와 같은 클라우드 서비스를 이용하며, 개발자는 필요한 만큼 서버와 데이터베이스 공간을 빌려 사용한다.

### 1. 클라우드 전용 DB 서비스(AWS RDS 등) 활용

실제 서비스 운영에서 가장 일반적으로 사용하는 방식이다.

클라우드 제공 업체가 관리하는 **데이터베이스 전용 서버**를 빌려 사용한다.
백엔드 서버(Spring Boot)는 이 데이터베이스와 연결되어 사용자가 작성한 데이터를 저장하고 조회한다.

예를 들어 사용자가 회고 내용을 작성하면:

1. 사용자가 웹 서비스에서 글 작성
2. Spring Boot 백엔드 서버가 요청 전달
3. AWS RDS에 있는 MySQL 데이터베이스에 저장
4. 이후 사용자가 다시 조회하면 DB에서 데이터를 가져와 화면에 표시

하는 구조이다.

이 방식은 데이터베이스 백업, 장애 관리, 성능 관리 등을 클라우드 서비스에서 지원하기 때문에 실제 서비스 운영에서 많이 사용된다.

---

### 2. 가상 서버(AWS EC2 등)에 직접 설치하는 방식

하나의 클라우드 컴퓨터를 빌린 뒤, 그 안에 필요한 프로그램을 직접 설치하는 방식이다.

예를 들어 하나의 EC2 서버 안에:

* Spring Boot 백엔드 서버
* MySQL 데이터베이스

를 같이 설치하여 운영한다.

이 방식은 별도의 DB 서버 비용이 들지 않아 저렴하다는 장점이 있다.
따라서 사용자 수가 많지 않은 초기 서비스(MVP)나 개인 프로젝트에서 많이 활용된다.

하지만 사용자가 증가하면 서버 자원을 나눠 써야 하기 때문에 성능 관리와 확장이 어렵다는 단점이 있다.

# 대용량 데이터 적재 및 비용 문제 해결 방식

사용자가 증가하면 “데이터가 계속 쌓이는데 데이터베이스가 터지지 않을까?”라는 문제가 생길 수 있다.

하지만 서비스 구조를 적절하게 설계하면 많은 사용자가 이용해도 안정적으로 운영할 수 있다.

---

### 1. 텍스트 데이터 중심 구조

KPT 서비스에서 저장하는 데이터는 대부분 회고 내용, 제목, 댓글과 같은 **텍스트 데이터**이다.

텍스트 데이터는 이미지나 영상처럼 큰 용량을 차지하지 않는다.

예를 들어:

* 짧은 글 수천만 건 → 수 GB 수준
* 사진, 영상 수천만 건 → 수 TB 이상 가능

처럼 저장 용량 차이가 크다.

따라서 초기 서비스에서는 고성능 서버가 아니더라도 충분히 운영할 수 있다.

---

### 2. 이미지와 같은 대용량 파일은 별도 저장

추후 사용자가 회고와 함께 사진을 첨부하는 기능이 추가되더라도 이미지를 데이터베이스(MySQL)에 직접 저장하지 않는다.

이미지 파일은 **AWS S3와 같은 파일 저장 전용 서비스**에 저장한다.

저장 방식은 다음과 같다.

1. 사용자가 사진 업로드
2. 사진 파일 → AWS S3 저장
3. S3에서 생성된 이미지 주소(URL)를 MySQL에 저장

즉 MySQL에는 이미지 자체가 아니라:

```
https://s3.amazonaws.com/example/image1.png
```

같은 링크 정보만 저장한다.

이렇게 하면 데이터베이스의 크기가 불필요하게 커지는 것을 막을 수 있고, 이미지 관리도 효율적으로 할 수 있다.

---

### 3. 사용자별 데이터 분리 관리

하나의 데이터베이스에는 여러 사용자의 데이터가 함께 저장된다.

하지만 데이터가 섞여서 보이는 것은 아니다.

테이블 설계 시 각 데이터에는 `user_id`라는 사용자 식별 값을 함께 저장한다.

예:

| id | user_id | content |
| -- | ------- | ------- |
| 1  | 10      | 오늘 배운 점 |
| 2  | 20      | 프로젝트 회고 |
| 3  | 10      | 개선할 점   |

사용자가 자신의 데이터를 요청하면 백엔드에서는:

```sql
WHERE user_id = 현재 로그인한 사용자 ID
```

조건을 사용한다.

따라서 같은 데이터베이스 안에 여러 사용자의 정보가 존재하더라도, 로그인한 사용자에게 해당하는 데이터만 조회된다.
