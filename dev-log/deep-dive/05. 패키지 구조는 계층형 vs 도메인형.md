# 고민 주제

> **Spring 프로젝트 패키지를 계층형으로 나눌까, 도메인형으로 나눌까?**

---

# 결론

**도메인형**으로 결정. `user/`, `retrospective/`, `kpt/`처럼 기능별 폴더 안에 그 도메인의 Entity/Repository/Service/Controller를 다 넣는다.

---

# 배경

Entity 코드를 짜기 시작하면서, 이걸 어떤 폴더 구조로 나눌지부터 정해야 했음.

---

# 두 방식 비교

**계층형**: `entity/`, `repository/`, `service/`, `controller/` 폴더로 나누고, 그 안에 모든 도메인이 섞여 있음.

```
entity/
  User.java
  Retrospective.java
  KptItem.java
  ...
repository/
  UserRepository.java
  RetrospectiveRepository.java
  ...
```

**도메인형**: `user/`, `retrospective/`, `kpt/`처럼 기능별 폴더 안에 그 도메인의 Entity/Repository/Service/Controller를 다 넣음.

```
user/
  User.java
  UserRepository.java
  UserService.java
  UserController.java
retrospective/
  Retrospective.java
  RetrospectiveRepository.java
  ...
```

---

# 왜 도메인형인가

Entity가 벌써 9개(User, Retrospective, KptItem, KptNote, AiRecommendation, WeeklySummary, Summary, ProblemPattern, ProblemPatternTry)라, 계층형으로 가면 `entity/` 폴더 하나에 9개가 다 섞여서 나중에 찾기 불편해짐. 도메인형은 `retrospective/` 폴더 하나만 열면 그 기능 관련 코드가 다 모여있어서, 기능 단위로 이해하거나 설명하기(포트폴리오 면접 등)도 더 수월함.

**트레이드오프**: 도메인 간에 서로 참조할 때(예: Retrospective가 User를 참조) 패키지를 넘나들어야 해서, 계층형보다 처음엔 조금 더 복잡하게 느껴질 수 있음.

---

# 한 줄 정리

> **Entity 개수가 많아지고 기능 단위로 이해해야 할 필요가 클수록, 도메인형 패키지 구조가 유리하다.**
