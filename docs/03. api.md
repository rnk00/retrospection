# API 설계 (Retrospective AI)

docs/erd.md에서 확정한 스키마를 기준으로, 화면 단위로 하나씩 짚어가며 설계함. 여기서도 erd.md처럼 "처음 생각 → 고민 → 최종 결정" 흐름을 남김.

---

## 0. 공통 규칙

### 인증

로그인 토큰(JWT)은 **요청 헤더**로 전달한다. `user_id`를 URL이나 body에 직접 넣지 않는 이유는, 그러면 값만 바꿔서 다른 사람 정보에 접근할 수 있게 되는 보안 문제 때문. 서버는 헤더의 토큰을 까서 누구의 요청인지 판단한다.

콜백 API(`/auth/kakao/callback`, `/auth/github/callback`)처럼 **로그인 자체를 하는 API**만 예외 — 이때는 아직 토큰이 없으므로 응답으로 토큰을 발급해서 body에 담아 돌려준다. (이후부터는 그 토큰을 헤더에 실어 보냄)

### 파라미터: 쿼리 vs 경로

- **쿼리 파라미터(`?key=value`)**: "이 조건에 맞는 걸 다 보여줘" — 결과가 여러 개 나올 수 있는 **필터링**. 예: `GET /retrospectives?year=&month=`
- **경로 파라미터(`/{값}`)**: "바로 이거 하나를 가리켜" — 결과가 정확히 하나인 **특정 자원 식별**. 예: `GET /retrospectives/{date}`

헷갈렸던 사례: `GET /retrospectives/{date}`도 처음엔 쿼리로 하려다가, `date`가 유니크 제약(`user_id, date`)으로 항상 하나만 가리키는 값이라 경로가 맞다고 정정함. 반대로 `GET /retrospectives?year=&month=`을 경로로 바꾸려던 시도는, 이건 "그 달의 여러 회고 중 필터링"이라 쿼리가 맞다고 다시 정리함.

또 하나: `POST` 요청에 값을 넣을 때도 쿼리 파라미터로 넣으려다가, "쿼리는 필터링용, POST의 입력값은 body"라는 기준으로 통일함 (`POST /summaries?period_start=&period_end=` → `POST /summaries` body `{period_start, period_end}`로 정정).

### body에 넣는 기준

body에는 **서버가 스스로 알아낼 수 없는, 클라이언트가 새로 입력한 값**만 담는다. `user_id`(토큰으로 판단), 회고 내용(서버가 DB에서 직접 조회 가능) 같은 건 아무리 관련 있어도 body에 다시 넣지 않는다. 예: `POST /retrospectives/{date}/github`는 이미 DB에 있는 회고를 GitHub로 올리는 것뿐이라 body가 필요 없음.

### 상태 코드

- **200 OK**: 성공, DB에 새로 생성된 건 없음 (조회, 미리보기, 수정)
- **201 Created**: 성공, DB에 새 리소스가 생성됨
- **204 No Content**: 성공, 돌려줄 데이터가 없음 (삭제, 저장 안 되는 액션)
- **429 Too Many Requests**: rate limit 초과 (AI 생성류 API에 적용 예정, 구체적 제한 횟수는 구현 시 결정)

### method 선택 기준

- **GET**: 조회
- **POST**: 생성 (매번 새 리소스가 쌓임) — 또는 저장하지 않는 계산/액션
- **PUT**: 이미 있는 값을 전체 교체 (덮어쓰기, 재호출해도 결과 동일 = idempotent)
- **PATCH**: 검토했으나 결국 안 씀 — "바뀐 필드만 보낸다"가 원래 의미인데, 우리 폼은 항상 전체 값을 다시 보내는 구조라 그 의미에 안 맞음. 그래서 회고 수정도 PATCH가 아니라 PUT으로 감.
- **DELETE**: 삭제

## 1. 인증 (Kakao / GitHub 로그인)

```
POST /auth/kakao/callback   body {code} → {token}
POST /auth/github/callback  body {code} → {token}
```

### 고민한 지점

**"로그인 시작" API는 안 만듦.** 카카오/GitHub 로그인 페이지로 이동시키는 URL에는 `client_id`, `redirect_uri`처럼 이미 공개된 값만 들어가서, 프론트가 그 URL을 직접 만들어 리다이렉트하면 됨. 우리 서버를 거칠 이유가 없어서, 콜백(code 받는 시점) API만 만듦.

**code / access_token / 우리 토큰, 세 가지를 구분함.**
1. `code` — provider가 리다이렉트로 한 번만 주는 인가 코드. 이것만으론 아무것도 못 함.
2. `access_token` — 우리 서버가 `code` + `client_secret`으로 provider와 교환해서 받는 값. **프론트는 절대 못 봄.**
3. **우리가 발급하는 로그인 토큰(JWT)** — 위 과정이 끝나면 서버가 새로 만들어서 프론트에 주는 것. 이후 모든 요청의 헤더에 실림.

`access_token`을 URL에 그대로 노출하지 않고 `code`를 거치는 이유: `access_token`은 그 자체로 API 호출이 가능한 민감한 값이라 브라우저 주소창/히스토리에 노출되면 위험함. `code`는 한 번 쓰고 버려지고 `client_secret` 없이는 아무것도 못 하니 노출돼도 상대적으로 안전함.

**GitHub는 로그인과 연동이 완전히 별개.** GitHub도 카카오처럼 독립 로그인 버튼으로 씀. 이 경우 "카카오로 가입한 사람이 GitHub로도 로그인하면 계정이 두 개로 쪼개지는" 문제가 있는데, 이메일 매칭 등으로 자동 병합하지 않고 **그냥 별개 계정으로 취급**하기로 함 (자세한 이유는 erd.md §1-1 참고). GitHub 저장소 연동(자동 커밋)은 이 로그인과 전혀 다른 기능이라 아래 §2로 분리.

**콜백 처리 로직이 단순해짐.** 애초에 스키마에 `OAuthAccount`를 따로 뒀던 이유가 "한 유저가 카카오+GitHub를 동시에 연동할 수 있어야 해서"였는데, 계정 병합을 안 하기로 하면서 그 시나리오 자체가 없어짐 — 유저는 평생 정확히 하나의 `(provider, provider_user_id)`만 가지므로, `OAuthAccount` 테이블을 없애고 `provider`/`provider_user_id`를 `User`에 직접 통합함 (자세한 이유는 erd.md §1-1 참고). 그래서 콜백 로직은 "`(provider, provider_user_id)`로 `User`를 조회, 없으면 새로 생성"이라는 한 단계로 끝남.

## 2. GitHub 연동 (PAT)

```
PUT /github/integration   body {pat, owner, repo} → 200 {pat, owner, repo}
GET /github/integration   → 200 {owner, repo, github_upload_enabled}
```

### 고민한 지점

**처음엔 OAuth 방식(§1과 동일하게 code/callback)으로 설계했었음.** 근데 이건 "로그인 확인"을 넘어서 서버가 사용자 대신 GitHub API를 호출(커밋)해야 하는 거라, 매번 access_token을 갱신/관리해야 하는 부담이 있음. **PAT(Personal Access Token)를 사용자가 GitHub에서 직접 발급받아 붙여넣는 방식**으로 바꿔서, OAuth 흐름 자체가 필요 없어짐 (redirect, code 교환 없이 그냥 값 저장).

**이름을 `/github/pat`에서 `/github/integration`으로 바꿈.** body에 `pat`뿐 아니라 `owner`, `repo`도 같이 들어가는데, 경로 이름이 `pat`이면 그것만 다루는 것처럼 오해할 수 있어서, erd.md에서 쓰던 개념 이름(`GithubIntegration`)을 그대로 가져옴.

**method는 PUT.** 유저당 설정값이 하나뿐이고(User와 1:1), 재저장하면 항상 기존 값을 덮어쓰는 구조라 POST가 아니라 PUT.

**조회 응답에는 `pat` 자체를 넣지 않음.** 비밀 값을 그대로 돌려주는 건 보안상 좋지 않아서, 화면에 필요한 `owner`/`repo`/`github_upload_enabled`만 내려줌. "연동 여부"를 나타내는 별도 플래그도 안 둠 — `owner`/`repo`가 null이면 그 자체로 "연동 안 됨"을 알 수 있음 (§4의 `written` 필드를 뺀 것과 같은 이유).

**레포지토리 변경 시 기존 파일은 그대로 둠.** `owner`/`repo`가 바뀌어도 이전 레포에 이미 올라간 파일을 새 레포로 옮기거나 복사하지 않음 — 이후 업로드부터만 새 레포에 반영됨. 대신 `owner`/`repo`가 하나라도 바뀌면, 그 유저의 모든 `retrospective.is_github_synced`를 `false`로 일괄 초기화함 (새 레포엔 아직 파일이 없으므로 "동기화됨" 상태가 더 이상 유효하지 않음).

## 3. 내 정보

```
GET    /users/me → {nickname}
PATCH  /users/me body {nickname} → 200 {nickname}
DELETE /users/me → 204
```

설정 화면/헤더에 닉네임 표시용. URL에 `user_id`를 안 넣고 `me`로 쓰는 이유는 §0 인증 규칙과 동일 — 토큰으로 이미 누구인지 판단하므로.

### 고민한 지점

**닉네임 수정은 필드 하나뿐이라 PATCH를 씀.** §4와 다르게 폼 전체를 다시 보내는 구조가 아니라 값 하나만 부분 수정하는 거라, 여기서는 PATCH의 원래 의미("바뀐 필드만 보낸다")에 그대로 맞음.

**계정 삭제는 GitHub에 아무 요청도 안 보냄.** "GitHub에는 흔적을 남기지 않고 DB만 삭제한다"는 정책이라, `user` 행 하나만 지우면 됨 — `retrospective`, `weekly_summary`, `kpt_note` 등 자식 테이블은 전부 `ON DELETE CASCADE`로 걸려있어서 자동으로 같이 삭제됨. 별도로 각 테이블을 순회하며 지우는 로직이 필요 없음.

## 4. 회고 (Retrospective)

```
GET    /retrospectives?year=&month=              → {monthly_count, monthly_avg_score, retrospectives: [{date, score}]}
GET    /retrospectives/{date}                     → {id, score, is_github_synced, keep: [KptItem], problem: [KptItem], try: [KptItem]}
POST   /retrospectives                            body {date, score} → 201 {id, date, score}
PUT    /retrospectives/{date}                     body {score} → 200 {score} (is_github_synced를 false로 되돌림)
DELETE /retrospectives/{date}?delete_github=       → 204
POST   /retrospectives/{date}/github              → 204 (is_github_synced를 true로 변경)
```

(`KptItem` 모양은 §4-1 참고)

### 고민한 지점

**캘린더 조회 응답이 배열에서 객체로 바뀜.** 원래는 `[{date, score}]` 배열만 줬는데, 캘린더 화면에 "이번 달 통계"(작성 개수, 평균 점수)도 같이 떠야 해서, 이 두 값을 배열 앞에 얹어야 함. 배열 하나만 덜렁 주면 이 두 값을 넣을 자리가 없어서, `{monthly_count, monthly_avg_score, retrospectives: [...]}` 형태로 감쌈. `written` 필드는 여전히 안 넣음 — 회고 쓴 날짜만 배열에 담기로 했으므로 배열에 있다는 사실 자체가 "작성함"을 의미.

**KPT 구조 변경으로 회고 API의 역할이 축소됨.** Keep/Problem/Try가 텍스트 컬럼에서 `KptItem`(항목별 리스트)으로 분리되면서(erd.md §1-2 참고), 회고 API는 이제 `date`, `score`만 다루고 KPT 내용은 전부 §4-1의 KptItem API가 담당함.
- `POST /retrospectives`: 그 날짜의 회고를 새로 만드는 것뿐이라 body에 `score`만 있으면 됨 (기본값 3이라 생략 가능하게 할 수도 있지만, 명시적으로 받기로 함).
- `PUT /retrospectives/{date}`: 이제 `score` 하나만 바꾸는 API가 됨. 필드가 하나뿐이라 PATCH로 바꿀지 고민했으나, 값 하나를 통째로 교체하는 것도 여전히 PUT의 정의(전체 교체, idempotent)에 맞고, 앞으로 회고 레벨에 필드가 늘어날 가능성도 있어 PUT을 유지함.
- `GET /retrospectives/{date}` 응답의 `keep`/`problem`/`try`는 이제 텍스트가 아니라 `KptItem` 배열. `id`는 여전히 필요함(§4-1, ai-recommendations에서 사용).

**14일 수정 가능 기간을 서버에서 검증함.** `POST`/`PUT`(그리고 KptItem의 생성/수정)은 `date`가 `[오늘-14일, 오늘]` 범위 밖이면 `403`으로 거부. 미래 날짜와 14일 이전 날짜는 에러 메시지가 다름:
- 미래 날짜: `403 {"message": "미래의 회고는 작성할 수 없습니다."}`
- 14일 이전 날짜(신규 작성 시): `403 {"message": "14일 이전 회고는 작성할 수 없습니다."}`

`DELETE`는 이 제한에서 예외 — "읽기 전용"이 되어도 삭제는 항상 가능하게 하기로 함. 프론트에서도 버튼을 숨기지만, 서버가 직접 호출을 막아야 우회를 방지할 수 있어서 서버 검증이 필수.

**`DELETE`에 `delete_github` 쿼리 파라미터를 추가함.** 회고를 지울 때 이미 GitHub에 업로드된 마크다운 파일까지 같이 지울지는 정책으로 미리 정하지 않고, 삭제 시점에 사용자에게 직접 물어보기로 함 — 그 선택을 `?delete_github=true|false`로 전달.

**`POST /retrospectives/{date}/github`(자동 커밋) body는 없음.** 서버가 이미 `date` + 토큰만으로 DB에서 내용을 조회할 수 있어서, 프론트가 내용을 다시 보내줄 필요가 없음.

**`is_github_synced`로 DB-GitHub 동기화 상태를 표시함.** 업로드 성공 시 `true`로, 점수 변경/KPT 항목 변경(§4-1의 생성·수정·삭제·순서변경) 시 `false`로 되돌림. `updated_at` 비교 대신 별도 boolean을 쓰는 이유는 erd.md §1-2 참고 — KPT 내용이 `KptItem`(별도 테이블)에 있어서 `Retrospective.updated_at`만으로는 KPT 수정을 감지할 수 없기 때문.

## 4-1. KPT 항목 (KptItem)

```
POST /retrospectives/{date}/kpt-items       body {type, content} → 201 {id, type, content, sequence_order}
POST /retrospectives/{date}/kpt-items/bulk  body {type, contents: [text, text, ...]} → 201 [KptItem, KptItem, ...]
PUT  /retrospectives/{date}/kpt-items/{id}  body {content} → 200 {...} 또는 (빈 값이면) 204
DELETE /retrospectives/{date}/kpt-items/{id} → 204
PUT  /retrospectives/{date}/kpt-items/order body {type, order: [id, id, id]} → 204
```

### 고민한 지점

**KPT를 "항목 단위 API"로 새로 만든 이유.** Keep/Problem/Try가 각각 최대 20개 항목을 가지고, 항목 하나 수정 시 1초 debounce 자동저장, 드래그로 순서 변경까지 지원해야 해서, 회고 전체를 한 번에 `PUT`하는 기존 방식으로는 표현이 안 됨(자세한 배경은 erd.md §1-2). 그래서 항목 하나하나를 독립적으로 CRUD할 수 있는 서브 리소스로 뺌.

**생성 시 `sequence_order`는 body에 없음.** 새 항목은 항상 그 type의 맨 뒤에 추가되는 거라, 서버가 "현재 그 type의 최대 order + 1"로 알아서 계산함. 프론트가 순서를 계산해서 보낼 필요가 없음.

**수정 시 내용이 비면 삭제로 처리.** "빈 항목은 저장하지 않는다"는 정책 때문에, `PUT`으로 `content`를 지우면(자동저장으로 빈칸이 되면) 서버가 그 항목을 그냥 지워버리고 `204`를 반환함 — 프론트가 "지울지 저장할지"를 따로 판단할 필요 없이 항상 `PUT`만 호출하면 됨.

**순서 변경은 배열 하나로 한 번에 처리.** 드래그로 순서를 바꾸면 그 type 안의 항목들이 통째로 재배열되는 거라, 항목 하나씩 순서를 바꾸는 API 여러 번 호출하는 대신, 새 순서대로 정렬된 id 배열을 한 번에 보내서 서버가 순서를 1부터 다시 매김.

**타입/개수/기간 제한은 이 API들이 담당.** `type`별 최대 20개 초과, 14일 수정 기간(§4 참고)은 모두 이 API(생성/수정)에서 검증함.

**`bulk`는 KPT 메모(KptNote) 불러오기 전용.** KPT 메모 모달에서 여러 개를 체크해서 한 번에 불러올 수 있어야 하는데, 하나씩 `POST`를 여러 번 호출하면 20개 제한 체크가 중간에 애매해질 수 있어서, 선택한 내용들을 배열로 한 번에 보내 서버가 한 번에 검증·생성하도록 함.

**이 API들도 전부 `is_github_synced`를 `false`로 되돌림.** 생성/수정(삭제 처리 포함)/삭제/순서변경 어느 쪽이든 그 회고의 GitHub 업로드 내용과 달라지는 거라, §4의 동기화 상태 관리 대상에 포함됨.

## 4-2. 홈 화면 요약 (최근 회고 / streak)

```
GET /home-summary → {recent: [{date, score}], streak}
```

### 고민한 지점

**§4의 "이번 달 통계"랑 분리된 이유.** "이번 달 개수/평균 점수"는 캘린더가 지금 보고 있는 `year`/`month` 범위에 그대로 종속되는 값이라 `GET /retrospectives`에 얹었는데(§4), "최근 회고 3개"와 "streak"는 캘린더에서 몇 월을 보고 있든 항상 똑같은 값(실제 최신 기준)이라 그 요청과는 무관함. 그래서 별도 엔드포인트로 뺌.

**streak는 서버가 계산해서 숫자로 줌.** §8 통계 대시보드는 기간 전체의 일별 데이터를 다 내려주니 프론트가 그 배열로 직접 streak를 계산할 수 있었는데, 이 API는 "최근 3개"만 주고 전체 기록을 안 주기 때문에 프론트가 계산할 재료가 없음. 그래서 여기서는 예외적으로 서버가 (오늘부터 거꾸로 훑어 끊기는 지점을 찾는 방식으로) 계산해서 `streak` 숫자만 내려줌.

## 4-3. KPT 메모 (KptNote)

```
POST   /kpt-notes             body {type, content} → 201 {id, type, content, sequence_order}
GET    /kpt-notes             → {keep: [KptNote], problem: [KptNote], try: [KptNote]}
PUT    /kpt-notes/{id}        body {content} → 200 {...} 또는 (빈 값이면) 204
DELETE /kpt-notes/{id}        → 204
PUT    /kpt-notes/order       body {type, order: [id, id, id]} → 204
```

### 고민한 지점

**§4-1의 KptItem API를 그대로 재사용한 패턴.** 날짜/회고에 종속되지 않고 `user_id` 기준으로 쌓인다는 것만 다르고, type별 최대 20개·빈 값 미저장·순서 변경 방식은 KptItem과 동일함(erd.md §2-4 참고).

**"메모에 추가" 버튼은 이 `POST /kpt-notes`를 그대로 호출.** KPT 작성 화면의 개별 항목이든, 분석 탭의 AI Try 추천이든, "메모에 추가"를 누르면 그 항목의 `content`만 복사해서 `POST /kpt-notes`로 보냄 — 새 API 없이 기존 생성 API 재사용 (AI Try 추천의 "추가" 버튼이 `POST /kpt-items`를 재사용했던 것과 같은 패턴).

**`GET /kpt-notes` 응답은 배열이 아니라 type별 객체.** 화면에서 Keep/Problem/Try 세 목록을 동시에 보여줘야 해서, 한 번의 요청으로 세 개 다 받도록 함.

## 5. AI 추천 (AiRecommendation)

```
POST /retrospectives/{date}/ai-recommendations → 201 {id, suggested_try, created_at, remaining_today}
GET  /retrospectives/{date}/ai-recommendations → {remaining_today, recommendations: [{id, suggested_try, created_at}]}
```

### 고민한 지점

**경로는 복수형(`ai-recommendations`).** 회고 하나에 여러 추천이 쌓이는 히스토리 구조(1:N)라, 컬렉션에 새 항목을 추가하는 의미의 복수형이 맞음.

**생성 응답에 `id`가 필요함.** 처음엔 `suggested_try`, `created_at`만 생각했는데, 나중에 히스토리 목록에서 각 항목을 구분하려면 `id`가 있어야 해서 추가. 조회 응답도 같은 모양으로 통일.

**추천을 "추가"해도 나머지 추천은 안 지움.** Try가 KptItem 리스트로 바뀌면서(§4-1 참고), 추천 하나를 "추가"하는 것도 별도 API 없이 `POST /retrospectives/{date}/kpt-items`를 `{type: "TRY", content: <추천 텍스트>}`로 호출하는 것뿐임 — 사용자가 Try를 직접 타이핑해서 추가하는 것과 서버 입장에서 동일한 동작. 그래서 서버는 "이게 AI 추천에서 온 건지"조차 구분하지 않고, 선택 안 된 나머지 추천 히스토리도 그대로 둠. 하루 생성 제한(10회) 덕분에 회고 하나당 최대 30개(10회 × 3개) 남짓만 쌓이고 텍스트도 짧아서, 굳이 지울 이유가 없다고 판단(자세한 계산은 erd.md §2-2 참고).

**남용 방지.** 매번 AI API 호출 비용이 들고 히스토리도 쌓이므로, 구현 시 사용자당 rate limit(초과 시 `429`)을 걸기로 함. 구체적인 횟수는 구현 시점에 결정.

**남은 횟수를 응답에 항상 포함(`remaining_today`).** "오늘 추천 가능 횟수 7/10" 같은 걸 화면에 상시 보여줘야 해서, 조회(`GET`) 응답에도 남은 횟수를 넣어 배열을 객체로 감쌈(§4 캘린더 조회에 `monthly_count`를 얹었던 것과 같은 방식). 생성(`POST`) 응답에도 갱신된 값을 같이 줘서, 생성 직후 다시 조회하지 않아도 화면 숫자를 바로 갱신할 수 있게 함.

## 6. 주간 요약 (WeeklySummary, 홈 화면)

```
PUT /weekly-summaries/{week_start} → 200 {week_start, summary_content}
GET /weekly-summaries/{week_start} → 200 {week_start, summary_content} 또는 (아직 생성 전) 404
```

### 고민한 지점

**`week_end`는 요청/응답 어디에도 없음.** 항상 고정 7일 단위라 `week_start + 6일`로 계산 가능 — DB 컬럼도 같은 이유로 뺐음(erd.md §2-3 참고).

**기본값 처리.** API에서 `week_start`를 생략하면 서버가 자동으로 이번 주 월요일을 계산해 처리하는 방식도 고려하였다. 하지만 이번 주 월요일은 오늘 날짜만 알면 프론트엔드에서도 쉽게 계산할 수 있다. 따라서 서버가 별도의 기본값 처리 로직을 제공하기보다, 프론트엔드가 `week_start`를 항상 계산하여 명시적으로 전달하도록 결정하였다. 이 방식은 서버가 예외 상황을 처리할 필요가 없어 API가 더 단순하고 명확해진다.

**method는 PUT.** 재생성하면 기존 값을 덮어쓰는 구조(`UNIQUE(user_id, week_start)`)라 POST가 아니라 PUT.

**지난 주 넘겨보기도 같은 엔드포인트를 그대로 씀.** 캘린더 화면에서 지난 주로 넘기면 프론트가 그 주의 `week_start`로 `GET`을 호출 — 저장된 요약이 있으면 그대로 보여주고, `404`(한 번도 생성된 적 없는 주)면 같은 `week_start`로 `PUT`을 호출해 그 자리에서 생성함. "이번 주" 방문 시 자동 생성하던 것과 동일한 흐름이라 별도 엔드포인트가 필요 없음. 미래 주(`week_start`가 이번 주보다 미래)는 프론트에서 넘겨보기 대상에서 제외.

## 7. 요약 (Summary, 통계 화면) / 반복 문제 분석 (ProblemPattern)

두 기능이 구조적으로 완전히 같아서 같이 정리함.

```
POST   /summaries/preview          body {period_start, period_end} → 200 {summary_content}
POST   /summaries                  body {period_start, period_end, summary_content} → 201 {id, ...}
GET    /summaries                  → [{id, period_start, period_end, summary_content, created_at}]
DELETE /summaries/{id}             → 204

POST   /problem-patterns/preview   body {period_start, period_end} → 200 {problem, try: [text, text, text]}
POST   /problem-patterns           body {period_start, period_end, problem, try: [text, text, text]} → 201 {id, problem, try: [...], ...}
GET    /problem-patterns           → [{id, period_start, period_end, problem, try: [...], created_at}]
DELETE /problem-patterns/{id}      → 204
```

### 고민한 지점

**"미리보기 → 저장" 두 단계로 나눔.** 처음엔 WeeklySummary처럼 "생성 = 바로 저장(덮어쓰기)"으로 가려 했는데, 이 기능은 사용자가 기간을 자유롭게 골라 여러 번 만들어볼 수 있어서, 생성할 때마다 다 DB에 남기면 불필요하게 많이 쌓임. 그래서 생성 버튼은 결과만 보여주고(DB 저장 안 함), 사용자가 "저장"을 눌러야 진짜로 남게 함.

**저장 API는 미리보기 결과를 그대로 다시 보냄.** 저장할 때 AI를 다시 호출하지 않고, 프론트가 미리보기 응답으로 이미 받은 텍스트를 body에 그대로 실어 보내서 저장만 함 (AI 재호출 비용 절약).

**URL 이름 충돌을 겪고 재배치함.** 처음에 `POST /summaries`를 미리보기용으로 쓰려다가, "POST /컬렉션명은 보통 진짜로 저장한다"는 REST 관례와 어긋나서 (retrospectives POST는 실제로 저장되는데 얘만 안 되면 헷갈림), **저장 쪽에 `/summaries`(원래 이름)를 주고 미리보기 쪽을 `/summaries/preview`로 뺌.**

**UNIQUE 제약 없음, 개별 삭제로 정리.** 같은 기간을 여러 번 저장해도 되고, 대신 저장된 항목마다 `id`로 구분해서 개별 삭제 가능.

**ProblemPattern도 원래는 WeeklySummary처럼 "기간 고정 + 덮어쓰기"였다가, 실제로는 기간을 자유롭게 고르는 기능이라는 게 확인되면서 Summary와 동일한 정책으로 변경함.**

**`try`가 텍스트 하나에서 배열로 바뀜.** 분석 화면 재설계로 "기존 Try를 반복하지 않는 새로운 추천 여러 개"를 줘야 하게 되면서(화면엔 번호를 안 붙이지만 리스트임), `problem_pattern.try` 컬럼을 없애고 `ProblemPatternTry`로 분리함(erd.md §2-3 참고). API 응답/요청의 `try`는 이제 문자열 배열. `problem`(반복된 문제)은 여전히 텍스트 하나 그대로 유지.

## 8. 통계 대시보드

```
GET /statistics?period_start=&period_end= → {score_trend: [{date, score}]}
```

### 고민한 지점

**연속 작성일수(streak)를 서버가 계산해서 보내지 않음.** 처음엔 별도 필드로 계산해서 줄까 했는데, `score_trend`가 기간 전체의 일별 데이터를 다 담고 있어서, 프론트가 그 배열만으로 직접 계산할 수 있음 (오늘부터 거꾸로 훑으며 끊기는 지점 찾기). 서버가 계산할 이유가 없어져서 안 만듦.

**`unit`(일/주/월/연 그래프 단위) 파라미터는 API에 없음.** 처음엔 그래프 단위별로 데이터를 다르게(주별 평균, 월별 평균 등) 집계해서 줄까 했는데, 실제로는 "이번 주/이번 달/올해" 버튼이 `period_start`/`period_end` 범위만 바꾸는 것뿐이고 데이터는 항상 일별 그대로 보여주면 된다고 확인함. 그래서 `unit`은 프론트 전용 개념이고, API는 항상 일별 `{date, score}`만 내려줌.
