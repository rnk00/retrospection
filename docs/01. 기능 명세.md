# 기능 명세 (Retrospective AI)

ERD/API 설계 중 기획이 계속 흔들려서, 다시 화면/기능 단위로 구체적인 동작을 먼저 확정하기 위해 작성. (docs/erd.md, docs/api.md와 별개로 "무엇을 만드는지"만 정리하는 문서)

## 로그인

- GitHub 또는 카카오로 로그인
- 같은 사람이 GitHub로도, 카카오로도 로그인하면 **서로 다른 사용자로 인식**한다 (계정 병합 없음)
- 일반 로그인/회원가입은 없다 — 최초 로그인 시 자동으로 회원가입된다
- 로그인 상태 유지는 JWT로 한다
- 닉네임 기본값: GitHub는 GitHub username, 카카오는 카카오 프로필 이름

## 캘린더

- 미래 날짜는 클릭 불가
- 오늘 날짜는 강조 표시
- 과거는 **14일 전까지만 수정 또는 새로 작성 가능** (이 제한은 회고 자체뿐 아니라 KPT 항목 추가/수정에도 동일하게 적용됨), 그보다 이전은 **읽기 전용**(수정 관련 UI 자체가 노출되지 않음)
  - 단, **삭제는 기간 제한 없이 항상 가능**
- 회고는 하루에 하나만 작성 가능
- 날짜에는 점수에 맞는 색이 적용된다
- 날짜 선택 시
  - 이미 작성한 경우 → 작성한 내용을 볼 수 있고, 14일 이내 날짜면 수정 후 GitHub에 재업로드 가능 (같은 파일을 수정, 새 파일 안 생김). 14일이 지났으면 조회만 가능하고 수정 UI는 보이지 않음
  - 작성하지 않은 경우 → 14일 이내 날짜면 새로 작성 가능, 작성 후 GitHub 업로드 가능

### 분석

- 작성한 회고 개수, 평균 점수 — 캘린더가 보고 있는 달 기준
- 최근 회고 3개 (날짜, 점수) — 달 이동과 무관하게 항상 실제 최신 3개
- streak(연속 작성일수) — 오늘 안 쓰면 즉시 끊김, 달 이동과 무관
- 이번 주 AI 요약
  - 캘린더 화면 상단에 이번 주(월~일) 회고를 AI가 요약해서 보여준다
  - 캘린더 화면에 들어갔을 때, 이번 주 요약이 **아직 없으면 그때 한 번 자동 생성**하고, 이미 있으면 저장된 걸 그대로 보여준다 (방문할 때마다 재생성하지 않음)
  - 사용자가 "다시 요약하기"를 누르면 재생성하면서 기존 요약을 덮어씀 (히스토리 없음, 통계 화면의 "기간별 요약"과는 별개 기능, 재요약은 이번주만 가능, **하루 최대 3번**)
  - **지난 주들을 넘겨볼 수 있다** — 넘겨보다가 한 번도 생성된 적 없는 주를 만나면 그 자리에서 생성해서 보여준다 (미래 주는 넘겨보기 불가)

### KPT 메모 (Keep/Problem/Try)

- 날짜/회고와 무관하게 캘린더 화면에 독립적으로 쌓이는 메모 — 지키고 싶은 Keep, 반복되는 Problem, 하고 싶은 Try를 사용자가 직접 추가
- 캘린더 화면과 분석 탭(추가 버튼), kpt 작성 탭에서 모두 추가 가능
- AI는 관여하지 않는 순수 사용자 메모 (분석 탭의 AI 분석 콘텐츠와는 별개)
- 내용(content) 한 필드만 가짐 (날짜별 KPT 작성과 달리 이유/원인/규칙 같은 두 번째 필드 없음)
- K/P/T 각각 최대 20개까지, 드래그로 순서 변경 가능

## KPT 작성

캘린더에서 날짜 클릭 → 회고 작성 화면

- Keep: 유지하고 싶은 것
- Problem: 아쉬웠던 것
- Try: 다음에 시도할 것
- K/P/T 각각 최대 20개까지 작성 가능
- 드래그로 순서 변경 가능 (예: K1↔K2 순서를 바꾸면 화면에 보이는 번호도 그에 맞게 바뀜 — 번호 자체는 저장되는 값이 아니라 순서 기준으로 매번 계산해서 보여주는 것)
- 내용이 빈 값인 항목은 저장하지 않는다 (화면엔 빈 입력창이 있어도 DB엔 안 남음)
- 입력을 멈춘 뒤 1초 후 자동저장 (항목 단위 debounce)
- 항목이 이미 20개인 상태에서 추가를 시도하면, 저장하지 않고 "최대 20개까지 작성 가능합니다" 안내와 함께 거부
- 점수는 1~5, 기본값은 3점 (빈 값 허용 안 함)
- kpt 메모에서 불러오기 가능

### AI Try 추천

- 추천은 3개씩 나온다
- 다시 추천받을 수 있고, 히스토리는 계속 남긴다 (KPT를 수정/저장해도 히스토리는 지워지지 않음)
- 하루 10회로 제한 — **유저 한 명당** 하루 10회 (여러 날짜의 회고를 편집해도 합쳐서 10회. 특정 회고 단위 제한이 아님), KST 자정 기준 초기화
- 기존에 작성된 Try 내용은 참고하지 않고 추천을 생성한다
- 추천 텍스트에는 마크다운 문법 문자가 섞이지 않도록 한다
- "추가" 버튼을 누르면 그 추천이 새 Try 항목으로 맨 뒤에 추가된다 (Try가 이미 20개면 위 규칙대로 거부)
- 사용 모델: Google AI Studio의 모델 (모델명은 구현 시 결정)

### 현재 KPT 개선 버튼 (확장 단계)

- 지금 작성한 KPT 전체를 보고 개선 제안을 해주는 별도 기능 — MVP 이후 확장 단계에서 설계

### 저장 상태 표시

- **저장 전 / 저장 중 / 저장 완료 / 저장 실패** 상태를 표시한다
- 수동으로 저장할 수 있는 저장 버튼도 존재한다(누르면 캘린더 화면으로 넘어간다)
- 상태는 KPT 항목의 수정/추가/삭제/순서변경 + 점수(score) 변경까지 전부 포함해서, **회고 전체 기준**으로 관리한다 (항목 하나라도 안 끝났으면 "저장 완료"가 아님)
- 화면을 처음 열었을 때(아직 아무것도 안 고친 상태)는 "저장 완료"로 시작하지만, 색이 옅은 등의 ui적인 요소를 더한다
- 저장 실패 시 팝업으로 안내하고, 상태는 "저장 전"으로 되돌린다

### GitHub 업로드

- PAT로 인증한다
- 커밋 메시지는 일관된 형식으로 자동 생성 (예: `26-07-11 dot 회고`)
- 파일명은 `YY-MM-DD.md`
- 업로드 버튼은 **저장 상태가 "저장 완료"일 때만 활성화**된다 (저장 전/저장 중/실패 상태에서는 비활성화)
- PAT를 어떻게 저장할 것인지는 보안 쪽으로 공부가 필요함

## 분석 (통계 화면)

기간(period_start/period_end)을 사용자가 직접 지정. "내 회고를 숫자와 그래프로 보여주는 탭" — 아래 대시보드는 가공 없이 있는 그대로 시각화하고, AI 인사이트는 그걸 분석/제안하는 영역으로 구분.

### 대시보드

- **KPI**: 총 회고 수 / 평균 점수 / 현재 Streak / 이번 달 작성률(기간 내 작성한 날짜 ÷ 작성 가능한 날짜)
- **점수 변화 그래프**: 기간 내 날짜별 점수 추이. 기간 선택 가능, 그래프의 점을 클릭하면 해당 날짜 회고를 바로 열 수 있음
- **점수 분포**: 1~5점 각각 몇 번씩 줬는지 (예: 5점 32번, 4점 48번...) — "나는 보통 몇 점을 주는 사람인가"를 보여줌

### AI 인사이트

- **많이 언급된 Keep TOP 3**: 선택 기간 동안 자주 나온 Keep을 분석해서 보여줌 (예: "운동하기(15회)") — 비슷한 표현을 AI가 하나로 묶어서 횟수를 셈
- **많이 언급된 Problem TOP 3**: 위와 동일한 방식으로 Problem 분석
- **AI 기간별 분석**: 기간 내 회고를 종합해 자연어로 분석 (Keep/Problem 관계, 이전 대비 달라진 점 등 설명) — 기존 "기간별 회고 요약" 기능
- **AI 기간별 Try 추천**: 반복된 Problem을 기반으로 여러 개(①②③...)의 새로운 Try를 추천 — 기존 Try를 그대로 반복하지 않음. 기존 "기간별 P 기반 Try 추천" 기능이 단일 텍스트에서 **여러 개의 번호 매겨진 추천**으로 바뀜 (하루 KPT 작성 탭의 AI Try 추천처럼 여러 개 제공)
- 각 추천 Try 항목에도 KPT 메모 "추가" 버튼이 붙는다 (위 "KPT 메모" 참고)

### 기타

- KPT 메모(Keep/Problem/Try) 추가 버튼 — 캘린더/KPT 작성 화면의 KPT 메모와 같은 목록을 공유 (위 "KPT 메모" 참고)
- 요일별 분석(예: 매주 월요일 회고만 모아서 분석)은 **보류** — 나중에 다시 설계

## 설정

- 프로필: 닉네임 수정
- GitHub 연동: repository, PAT 입력 → 연동. PAT는 암호화하여 저장
  - PAT는 사실상 GitHub 비밀번호와 동급인 민감 정보로 취급한다 (유출되면 연동된 repo에 대해 읽기/쓰기/삭제/커밋까지 가능해짐) — 암호화 저장, 로그/응답/JWT에 절대 노출 금지, Fine-grained PAT(해당 repo만, Contents Read&Write 권한만) 사용을 안내
  - 저장 시 GitHub API로 즉시 유효성 검사 후 암호화 저장, 실패하면 "토큰이 유효하지 않습니다" 안내
  - PAT 입력란 근처에 **"PAT 만들기" 버튼**을 둔다 — GitHub의 Fine-grained PAT 생성 페이지로 새 탭 이동 + Repository/Permissions를 어떻게 설정해야 하는지 안내
  - PAT 등록(저장) 시 **경고 팝업**을 띄운다 — "Fine-grained PAT를 특정 레포 하나로 한정하고, Contents Read&Write 권한만 부여하세요. Classic PAT나 넓은 권한을 쓰면 유출 시 계정 전체(다른 프라이빗 레포 포함)가 위험해질 수 있습니다" 같은 내용. 사용자가 이 팝업을 확인해야 등록이 진행됨
- 알림 설정(일일 회고 알림, 주간 요약 알림, 시간 설정, 브라우저 push)은 **보류** — 나중에 다시 설계

## 계정 삭제

- User, 회고, KPT 항목 등 우리 DB의 모든 정보를 삭제한다
- GitHub 저장소의 파일은 그대로 둔다 (삭제 요청을 보내지 않음)

## 팀 기능 (확장, 추후 설계)

개인 회고는 "내가 회고를 잘 쓰는 것"이 목적이지만, 팀 회고는 "팀이 같은 문제를 공유하고 개선하는 것"이 목적이라 필요한 기능이 완전히 다름. 아직 상세 설계(ERD/API) 전이고 구현도 나중 — 여기는 브레인스토밍한 기능 후보만 정리해둠.

### 기능 후보 (우선순위 순, ★★★★★는 특히 중요)

1. **팀 생성 / 초대** (필수) — 팀 생성, 초대 링크/코드, 권한(Owner/Admin/Member). 개인 계정 아래에 여러 팀(예: SSAFY 9반, 공통프로젝트, 토이프로젝트)을 만들 수 있는 구조
2. **회고 세션** (필수) — 팀 회고는 날짜가 아니라 세션(Sprint 1, Release 1.0, 프로젝트 종료 회고 등) 단위. 세션을 만들고 멤버들이 그 세션에 작성
3. **작성 마감** (필수) — 세션마다 마감 시각 설정, 마감 전까지만 수정 가능, 마감되면 자동 읽기 전용
4. **작성 여부 현황** (필수) — "6명 중 5명 작성" 식으로 팀장이 보기 좋은 현황 표시
5. **익명/실명 선택** (★★★★★) — 익명일 때 "리더가 일정관리를 잘 못했다" 같은 솔직한 의견이 더 잘 나옴
6. **공감(좋아요)** (★★★★★) — Problem/Try 항목에 공감, 공감 많은 순으로 정렬
7. **댓글** — 항목에 댓글로 토론
8. **AI 중복 의견 병합** (★★★★★) — "회의 길다/회의 너무 길다/회의 오래함" 같은 비슷한 의견을 AI가 하나로 묶어서 "18명 언급" 식으로 표시
9. **AI 요약** — 세션의 Keep/Problem/Try를 AI가 요약
10. **Try 투표** — 여러 Try 후보 중 팀이 투표로 채택할 Try 결정
11. **Try → Action Item** (★★★★★) — 채택된 Try를 담당자/기한/상태가 있는 Action Item으로 전환
12. **다음 회고 자동 연결** — 지난 세션의 Action Item을 다음 회고에서 완료/진행중/실패로 확인
13. **태그** — #회의, #기술, #일정, #협업 같은 태그
14. **AI 자동 분류** — React/Vue/Git처럼 사람들이 제각각 쓴 태그를 AI가 기술/협업/일정 등으로 자동 분류
15. **반복 문제 감지** — 여러 세션에 걸쳐 같은 문제(예: 회의)가 반복되면 AI가 감지해서 알려줌
16. **칭찬 카드** — "이번 Sprint MVP: 김철수, 코드리뷰 잘해줌" 같은 팀원 칭찬 카드
17. **회고 발표 모드** — 회의 때 Keep/Problem/Try를 전체 화면 슬라이드로 자동 진행
18. **AI 회고 진행자** — "가장 많이 나온 Problem은 PR 리뷰입니다" 식으로 AI가 회고 진행을 보조
19. **GitHub/Jira 연결** — Try를 Issue로 생성하고 완료 여부 연동
20. **개인 의견 공개 시점 조절** — 전원 제출 완료 전까지는 다른 사람 의견을 안 보여줌 (안 그러면 앞사람 의견에 끌려 씀)
21. **팀 KPI** — 회고 참여율, 평균 작성 시간, Try 완료율 등
22. **AI 갈등 감지** — "짜증/답답/소통" 같은 표현이 늘면 "커뮤니케이션 이슈 증가" 식으로 감지
23. **Best Keep** — 공감 가장 많은 Keep을 세션 하이라이트로 표시

### MVP 후보 (10개)

팀 생성 → 회고 세션 → 익명/실명 → 공감 → 댓글 → AI 중복 병합 → AI 요약 → Try 투표 → Action Item → 다음 회고에서 Action 자동 확인

이 정도만 있어도 "개인 KPT 앱"과는 확실히 다른 팀 회고 도구가 됨.

### 차별화 포인트

여러 팀 회고 도구가 "의견을 모으는 것"에는 집중하지만 "의견을 합의하고 실행하는 것"은 약한 경우가 많음. 그래서 다음 흐름이 가장 차별화될 수 있음:

**익명 회고 작성 → AI가 중복 의견 병합 → 팀이 공감으로 우선순위 결정 → Try 투표 → Action Item 생성 → 다음 회고에서 실행 여부 확인**

회고가 단순한 기록이 아니라 팀의 개선 프로세스 자체가 되는 방향. 특히 스프린트 단위로 일하는 프로젝트 팀/SSAFY 같은 곳에서 활용도가 높을 것으로 예상.

## 기타 (백로그, 우선순위 미정)

- 챗봇 (자소서 질문 대응용)
- 알림
- 회고 템플릿 (KPT 외 4F, Start/Stop/Continue 등)
- 통계: 작성률
- 검색
- 비슷한 회고 찾기
- commit 기반 자동 회고
- 크롬 확장 프로그램 -> 알람
- k,p,t 순서 변경 단축키로
- 점수 그래프에서 점 클릭하면 해당 회고 보여주기

## 우선순위

### MVP

- 소셜 로그인 (카카오/GitHub, 계정 병합 없음)
- 캘린더
- 날짜별 KPT 작성 (항목 단위, 최대 20개, 순서 변경, 저장 상태 표시)
- KPT 수정/삭제
- 회고 작성 여부 색상 표시

### 추가

- GitHub 업로드
- AI Try 추천
- 이번 주 AI 요약 (캘린더 화면, 지난 주 넘겨보기 포함)
- 통계 대시보드 (작성률, streak)
- 기간별 P 기반 Try 추천
- 기간별 회고 요약
