# 기능 명세 (Retrospective AI)

ERD/API 설계 중 기획이 계속 흔들려서, 다시 화면/기능 단위로 구체적인 동작을 먼저 확정하기 위해 작성. (docs/erd.md, docs/api.md와 별개로 "무엇을 만드는지"만 정리하는 문서)

## 로그인

- GitHub 또는 카카오로 로그인
- 같은 사람이 GitHub로도, 카카오로도 로그인하면 **서로 다른 사용자로 인식**한다 (계정 병합 없음)
- 일반 로그인/회원가입은 없다 — 최초 로그인 시 자동으로 회원가입된다
- 로그인 상태 유지는 JWT로 한다
- 닉네임 기본값: GitHub는 GitHub username, 카카오는 카카오 프로필 이름

## 캘린더

- 미래 날짜는 클릭 불가
- 오늘 날짜는 강조 표시
- 과거는 **14일 전까지만 수정 또는 새로 작성 가능** (이 제한은 회고 자체뿐 아니라 KPT 항목 추가/수정에도 동일하게 적용됨), 그보다 이전은 **읽기 전용**(수정 관련 UI 자체가 노출되지 않음)
  - 단, **삭제는 기간 제한 없이 항상 가능**
- 회고는 하루에 하나만 작성 가능
- 날짜에는 점수에 맞는 색이 적용된다
- 날짜 선택 시
  - 이미 작성한 경우 → 작성한 내용을 볼 수 있고, 14일 이내 날짜면 수정 후 GitHub에 재업로드 가능 (같은 파일을 수정, 새 파일 안 생김). 14일이 지났으면 조회만 가능하고 수정 UI는 보이지 않음
  - 작성하지 않은 경우 → 14일 이내 날짜면 새로 작성 가능, 작성 후 GitHub 업로드 가능

### 이번 달 통계

- 작성한 회고 개수, 평균 점수 — 캘린더가 보고 있는 달 기준
- 최근 회고 3개 (날짜, 점수) — 달 이동과 무관하게 항상 실제 최신 3개
- streak(연속 작성일수) — 오늘 안 쓰면 즉시 끊김, 달 이동과 무관

### 이번 주 AI 요약

- 캘린더 화면 상단에 이번 주(월~일) 회고를 AI가 요약해서 보여준다
- 캘린더 화면에 들어갔을 때, 이번 주 요약이 **아직 없으면 그때 한 번 자동 생성**하고, 이미 있으면 저장된 걸 그대로 보여준다 (방문할 때마다 재생성하지 않음)
- 사용자가 "다시 요약하기"를 누르면 재생성하면서 기존 요약을 덮어씀 (히스토리 없음, 통계 화면의 "기간별 요약"과는 별개 기능)

## KPT 작성

캘린더에서 날짜 클릭 → 회고 작성 화면

- Keep: 유지하고 싶은 것 + 이유
- Problem: 아쉬웠던 것 + 원인
- Try: 다음에 시도할 것 + 규칙
- K/P/T 각각 최대 20개까지 작성 가능
- 드래그로 순서 변경 가능 (예: K1↔K2 순서를 바꾸면 화면에 보이는 번호도 그에 맞게 바뀜 — 번호 자체는 저장되는 값이 아니라 순서 기준으로 매번 계산해서 보여주는 것)
- 내용/이유 둘 다 빈 값인 항목은 저장하지 않는다 (화면엔 빈 입력창이 있어도 DB엔 안 남음)
- 입력을 멈춘 뒤 1초 후 자동저장 (항목 단위 debounce)
- 항목이 이미 20개인 상태에서 추가를 시도하면, 저장하지 않고 "최대 20개까지 작성 가능합니다" 안내와 함께 거부
- 점수는 1~5, 기본값은 3점 (빈 값 허용 안 함)

### AI Try 추천

- 추천은 3개씩 나온다
- 다시 추천받을 수 있고, 히스토리는 계속 남긴다 (KPT를 수정/저장해도 히스토리는 지워지지 않음)
- 하루 10회로 제한 — **유저 한 명당** 하루 10회 (여러 날짜의 회고를 편집해도 합쳐서 10회. 특정 회고 단위 제한이 아님), KST 자정 기준 초기화
- 기존에 작성된 Try 내용은 참고하지 않고 추천을 생성한다
- 추천 텍스트에는 마크다운 문법 문자가 섞이지 않도록 한다
- "추가" 버튼을 누르면 그 추천이 새 Try 항목으로 맨 뒤에 추가된다 (Try가 이미 20개면 위 규칙대로 거부)
- 사용 모델: Google AI Studio의 모델 (모델명은 구현 시 결정)

### 현재 KPT 개선 버튼 (확장 단계)

- 지금 작성한 KPT 전체를 보고 개선 제안을 해주는 별도 기능 — MVP 이후 확장 단계에서 설계

### 저장 상태 표시

- 별도 "저장" 버튼은 없고, 대신 화면에 **저장 전 / 저장 중 / 저장 완료 / 저장 실패** 상태를 표시한다
- 상태는 KPT 항목의 수정/추가/삭제/순서변경 + 점수(score) 변경까지 전부 포함해서, **회고 전체 기준**으로 관리한다 (항목 하나라도 안 끝났으면 "저장 완료"가 아님)
- 화면을 처음 열었을 때(아직 아무것도 안 고친 상태)는 "저장 완료"로 시작한다
- 저장 실패 시 팝업으로 안내하고, 상태는 "저장 전"으로 되돌린다

### GitHub 업로드

- PAT로 인증한다
- 커밋 메시지는 일관된 형식으로 자동 생성 (예: `26-07-11 dot 회고`)
- 파일명은 `YY-MM-DD.md`
- 업로드 버튼은 **저장 상태가 "저장 완료"일 때만 활성화**된다 (저장 전/저장 중/실패 상태에서는 비활성화)

## 분석 (통계 화면)

- 기간별 P 기반 Try 추천 (사용자가 기간을 직접 지정)
- 기간별 회고 요약 (사용자가 기간을 직접 지정)
- 요일별 분석(예: 매주 월요일 회고만 모아서 분석)은 **보류** — 나중에 다시 설계

## 설정

- 프로필: 닉네임 수정
- GitHub 연동: repository, PAT 입력 → 연동. PAT는 암호화하여 저장
  - PAT는 사실상 GitHub 비밀번호와 동급인 민감 정보로 취급한다 (유출되면 연동된 repo에 대해 읽기/쓰기/삭제/커밋까지 가능해짐) — 암호화 저장, 로그/응답/JWT에 절대 노출 금지, Fine-grained PAT(해당 repo만, Contents Read&Write 권한만) 사용을 안내
  - 저장 시 GitHub API로 즉시 유효성 검사 후 암호화 저장, 실패하면 "토큰이 유효하지 않습니다" 안내
- 알림 설정(일일 회고 알림, 주간 요약 알림, 시간 설정, 브라우저 push)은 **보류** — 나중에 다시 설계

## 계정 삭제

- User, 회고, KPT 항목 등 우리 DB의 모든 정보를 삭제한다
- GitHub 저장소의 파일은 그대로 둔다 (삭제 요청을 보내지 않음)
